# Contributing

> Purpose: Define how changes should be proposed, validated, documented, and merged.
> When to update: When branch strategy, review expectations, commit conventions, or required checks change.
> Owner: {{OWNER_NAME}}.

## Contribution Flow

1. Open or reference the relevant issue, spec, or task packet.
2. Make the smallest reasonable change that solves the problem.
3. Update impacted docs in the same change.
4. Run the required validation steps before asking for review.

## Pull Request Expectations

- Explain the change clearly.
- Include validation evidence.
- Call out risk, rollback, and follow-up items.
- Link to the related planning docs when they exist.

## Documentation Expectations

- Update durable docs when behavior, architecture, or process changes.
- Do not leave placeholders or stale examples behind.

## Review Norms

- Prefer small PRs.
- Resolve ambiguity in docs, not only in chat.
- Surface assumptions and tradeoffs explicitly.
