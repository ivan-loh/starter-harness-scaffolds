# Decisions

> Purpose: Govern how architecture and process decisions are recorded and tracked over time.
> When to update: When ADR rules, statuses, or indexing conventions change.
> Owner: {{OWNER_NAME}}.

## When To Write An ADR

Write an ADR for long-lived decisions that materially affect architecture, operations, security, or delivery workflow.

## Status Model

- Proposed
- Accepted
- Superseded
- Deprecated

## Index

- `0000-template.md`: starter ADR template

## Rules

- Prefer one ADR per significant decision.
- Supersede old ADRs instead of silently rewriting history.
- Link the accepted ADR from the architecture or relevant operational docs.
